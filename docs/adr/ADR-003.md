# ADR-003: Elección de Dolibarr como ERP para la Fase de Demostración (Demo)

## Estado
Aceptado

## Contexto
El proyecto JGTravel requiere una infraestructura de ERP para gestionar operaciones turísticas, clientes y servicios. Durante la fase de investigación, evaluamos múltiples APIs y soluciones comerciales de mercado. Sin embargo, la gran mayoría de estas opciones conllevan costos de suscripción elevados o restricciones severas en sus capas gratuitas, lo que resulta inviable y poco eficiente para la etapa actual de demostración y validación del producto.

## Decisión
Hemos decidido adoptar **Dolibarr** como el sistema ERP central para la demo, desplegándolo en un **entorno de servidor local** controlado. Esta configuración servirá como un simulador y entorno de consulta para el desarrollo actual, estructurando la capa de integración de manera modular.

## Justificación

### 1. Viabilidad Económica y Mitigación de Costos en Fase de Demo
* **Open Source sin Restricciones:** Dolibarr nos permite acceder a un núcleo ERP completo de forma gratuita, eliminando la barrera económica de las APIs pagas durante la etapa de pruebas.
* **Control del Entorno:** Al montar un servidor local, el equipo tiene total autonomía para generar datos de prueba, modificar configuraciones y realizar peticiones de manera ilimitada sin temor a sobrepasar cuotas de uso o generar costos imprevistos.

### 2. Preparación para el Escalamiento (Abstracción de Integraciones)
* **Arquitectura de Transición Limpia:** El entorno local con Dolibarr actúa como un "banco de pruebas". Nos permite diseñar la lógica de sincronización y consumo de datos del frontend de manera agnóstica.
* **Futura Integración Plug-and-Play:** La estructura de consultas desarrollada hoy está pensada para que, el día de mañana, el cambio hacia otros sistemas pagos o plataformas líderes del mercado turístico se realice con el menor impacto posible. Solo se requerirá redirigir los conectores de la API sin tener que reescribir la lógica de negocio de JGTravel.

## Consecuencias

### Positivas
* **Cero Costos de Infraestructura:** Permite validar el flujo completo de la aplicación JGTravel de extremo a extremo sin invertir presupuesto en licencias de software en fases tempranas.
* **Flexibilidad de Datos:** Facilita la creación de escenarios complejos (rutas, reservas, perfiles de clientes) directamente en la base de datos local para robustecer la demostración.

### Negativas / Riesgos a Mitigar
* **Mantenimiento del Servidor Local:** El equipo de desarrollo debe hacerse cargo del despliegue, actualización y disponibilidad del servidor local.
  * *Mitigación:* Se utilizarán contenedores (como Docker) o scripts de configuración sencillos para garantizar que cualquier desarrollador pueda levantar el entorno Dolibarr con un solo comando.
* **Diferencias de API con Sistemas Propietarios:** Los endpoints y formatos de respuesta de Dolibarr no serán idénticos a las APIs pagas del futuro.
  * *Mitigación:* Se implementará una capa de abstracción (servicios/adaptadores) en nuestro código que normalice los datos recibidos de Dolibarr, asegurando que el resto de la aplicación consuma un formato estándar e intercambiable.

# ADR-003
## Modelo de Datos Propio

**Estado:** Aprobado

## Contexto

Los proveedores turísticos poseen estructuras de datos diferentes.

Depender directamente de dichos modelos acopla el sistema y dificulta futuras migraciones.

## Decisión

JGTravel utilizará un modelo de datos propio.

Ejemplo:

Hotel

Destino

Vuelo

Paquete

Reserva

Restaurante

Vehículo

Los datos provenientes de cualquier proveedor serán transformados antes de almacenarse o enviarse al frontend.

## Justificación

Aplicación del patrón Anti-Corruption Layer (ACL) de Domain-Driven Design.

## Consecuencias

El cambio de proveedor no implica modificaciones importantes en el frontend.
