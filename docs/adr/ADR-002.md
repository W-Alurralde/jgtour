# ADR-002: Elección de Firebase Authentication para el Ecosistema de Frontend (React y React Native)

## Estado
Aceptado

## Contexto
El proyecto requiere implementar un sistema de gestión de identidad y acceso (IAM) de forma ágil, segura y con soporte inmediato para inicios de sesión sociales (Google) y proveedores externos (Auth0). 

Inicialmente, la aplicación se desarrollará como una plataforma web utilizando **React**. Sin embargo, la hoja de ruta del producto contempla un **escalamiento a corto/mediano plazo hacia una aplicación móvil nativa utilizando React Native**. Necesitamos una solución de autenticación que minimice la duplicación de lógica y permita compartir la sesión del usuario entre ambas plataformas con el menor impacto arquitectónico posible.

## Decisión
Hemos decidido adoptar **Firebase Authentication** como el proveedor principal de identidad para el proyecto web en React, estructurando la capa de servicios de manera agnóstica para asegurar su compatibilidad directa con React Native en el futuro.

Para la integración con **Auth0** (si se requieren flujos corporativos avanzados), utilizaremos la funcionalidad de *Custom Tokens* de Firebase, manteniendo a Firebase como la puerta de entrada unificada para la base de datos y el resto de los servicios del frontend.

## Justificación

### 1. Velocidad de Lanzamiento (Time-to-Market)
* **SDKs Listos para Usar:** Firebase provee métodos nativos como `signInWithPopup` o `signInWithRedirect` que reducen el desarrollo de login con Google a un par de líneas de código.
* **Backend como Servicio (BaaS):** Evita la necesidad de construir, desplegar y mantener un microservicio de autenticación propio en esta etapa temprana.

### 2. Preparación para el Escalamiento (React Native Ready)
* **Ecosistema Compartido:** Firebase cuenta con la librería oficial `@react-native-firebase/auth`, la cual expone exactamente la misma firma de métodos y manejo de estados que el SDK web (`firebase/auth`).
* **Mismo Backend, Mismos Usuarios:** La base de usuarios creados en la web de React será idéntica y accesible inmediatamente desde la app móvil de React Native sin necesidad de migraciones de datos ni APIs puente.
* **Curva de Aprendizaje Cero:** El equipo de desarrollo utilizará los mismos conceptos (Tokens JWT, observadores como `onAuthStateChanged`) tanto en el desarrollo web como en el móvil.

### 3. Flexibilidad Arquitectónica (Puente con Auth0)
* **Consolidación de Sesión:** Al usar los tokens personalizados de Firebase para validar sesiones autenticadas previamente en Auth0, centralizamos las reglas de seguridad en un solo lugar. Esto aplica igual para web como para móvil.

## Consecuencias

### Positivas
* **Reducción de Deuda Técnica:** Al migrar a React Native, la lógica de negocio de autenticación (Contextos de React, hooks personalizados) se podrá reutilizar casi en un 90%.
* **Costos Operativos Bajos:** La capa gratuita de Firebase (Spark Plan) cubre hasta 50,000 usuarios activos mensuales, lo que mitiga costos en la etapa de validación de ambas plataformas.
* **Seguridad Delegada:** El almacenamiento seguro de credenciales, tokens de refresco y la infraestructura de criptografía quedan en manos de Google.

### Negativas / Riesgos a Mitigar
* **Vendor Lock-in (Dependencia del Proveedor):** Estamos acoplando el manejo de la sesión al ecosistema de Firebase. 
  * *Mitigación:* Se implementará un patrón de diseño de inversión de dependencias (crear un servicio de autenticación propio que envuelva al SDK de Firebase) para que, si en el futuro se decide cambiar de proveedor, solo se tenga que modificar ese archivo y no los componentes de React/React Native.
* **Diferencias de Flujo en Móvil:** El inicio de sesión con Google en React Native no puede usar pop-ups web; requerirá instalar la librería nativa de Google Sign-In e inyectar el token en Firebase. El diseño de la arquitectura debe prever esta separación de la capa de interfaz.
